  16     : 
  17     : to decompress a file, replacing it with the original uncompressed version:
  18     : 
  19     : ```bash
  20     : puncture -d /path/to/compressed_file.gz
  21     : ```

  24     : 
  25     : ```bash
  26     : puncture -cd /path/to/compressed_file.gz
  27     : ```
  28     : 
  29     : to decompress a file, keeping the original uncompressed version and specifying the output filename:
  30     : 
  31     : ```bash
  32     : puncture -cdk /path/to/compressed_file.gz > path/to/uncompressed_file
  33     : ```

  35     : ## philosophy
  36     : 
  37     : i wanted to create a project that tries to strike the middle ground between two extremes: it tries to be simple (but not the simplest) and fast (but not the fastest).
  38     : 
  39     : there are certainly simpler and shorter gzip implementations, but these projects are usually so stripped down, that they sacrifice educational value for simplicity. there isn't a lot to be learned from a 280 line program for someone who is already familiar with the topic.
  40     : 
  41     : there are also certainly faster, and more complicated implementations. these are usually so heavily optimized, that code readability suffers as a result. there _are_ things to be learnt from these projects as well, but their purpose is very different.
  42     : 
  43     : this project tries to be faster than naive implementations, but not so heavily optimized, that you need to spend hours to decipher what sort of bit-magic is happening. this way hopefully as many people as possible can take something away from this project.
  44     : 
  45     : since the primary goal of this project is to provide educational value, i tried to keep the codebase as simple as I could:
  46     : 
  47     : * [`src/bitreader.rs`](https://github.com/levyry/puncture/blob/main/src/bitreader.rs): a custom LSB-first bit reader. it maintains a `u128` internal cache to allow peeking and consuming a specific number of bits at a time.
  48     : * [`src/cached_writer.rs`](https://github.com/levyry/puncture/blob/main/src/cached_writer.rs): manages the split 32 KB sliding window history required by LZ77 and acts as a buffer, ensuring we do as few underlying `write` system calls as possible while also calculating the required CRC32 checksums.
  49     : * [`src/extraction.rs`](https://github.com/levyry/puncture/blob/main/src/extraction.rs): this is where the main DEFLATE algorithm lives: it decodes the dynamic Huffman trees, builds the lookup tables, and executes the main decompression loop. the GZIP header parsing also lives here.
  50     : * [`src/main.rs`](https://github.com/levyry/puncture/blob/main/src/main.rs): the CLI harness, responsible for calling the main DEFLATE algorithm with the correct inputs and flags.
  51     : 
  52     : i tried to use as few dependencies as possible:
  53     : 
  54     : * [`clap`](https://github.com/clap-rs/clap) for command line argument parsing
  55     : * [`crc32fast`](<https://github.com/srijs/rust-crc32fast>) for calculating the crc32 checksum
  56     :

  95     : ### codebase
  96     : 
  97     : the `gzip` binary uses the `zlib` library under the hood, so i will compare against that. the part of `zlib` responsible for decompression is roughly 2000 lines of code, while this implementation uses roughly 650 lines of code, which is a ~3x reduction in size.
  98     : 
  99     : additionally, this implementations uses some pretty high level abstractions for this low-level of a program (check out [`bitreader.rs`](https://github.com/levyry/puncture/blob/main/src/bitreader.rs)), and purposefully forgoes implementing some of the more hard to understand optimizations. there is no unsafe or any inline assembly.
 100     : 
 101     : ### complexity
 102     : 
 103     : * creates LUTs for dynamic huffman trees
 104     : * use a split 64 KB linear buffer for the 32 KB LZ77 sliding window and the writing buffer
 105     : * handles overlapping LZ77 matches using an exponential doubling algorithm
 106     : 
 107     : there are some optimizations i didn't end up implementing, such as:
 108     : 
 109     : * two-tier LUTs for huffman codes
 110     : * inlining the bit reading logic to extraction
 111     : 
 112     : with these implementations, the performance should be completely on-par with `gzip`.

   2     : //!
   3     : //! ## Constants
   4     : //!
   5     : //! RFC 1951 (henceforth: "the RFC") defines a lot of constants, tables, and
   6     : //! magic numbers. These are all defined at the beginning of the module.
   7     : //!
   8     : //! ## GZIP headers
   9     : //!
  10     : //! RFC 1952 defines the GZIP header format. I chose to ignore a lot of fields,
  11     : //! the only optional field I parse is the original file name.
  12     : //!
  13     : //! ## DEFLATE
  14     : //!
  15     : //! The DEFLATE algorithm is a bit-based compression/decompression algorithm.
  16     : //! It uses two main ideas for compression: Huffman coding, and the LZ77
  17     : //! algorithm.
  18     : //!
  19     : //! ### Huffman coding
  20     : //!
  21     : //! Huffman coding helps with assigning shorter bit-sequences to more
  22     : //! frequently occuring symbols. For example, in English, the letter 'e' is the
  23     : //! most frequently used letter on average, so it should be coded with a
  24     : //! shorter bit-sequence than, say, 'q' or 'v'.
  25     : //!
  26     : //! Huffman coding is a form of prefix coding, which is pretty similar to how
  27     : //! phone numbers work. For more insight, I recommend
  28     : //! [An Explanation of the Deflate Algorithm](https://zlib.net/feldspar.html)
  29     : //! by Antaeus Feldspar. The gist of it is that the symbols are arrenged in a
  30     : //! binary tree, where only the leaf nodes contain the symbols. To get the code
  31     : //! for any symbol, you travel from the root of the tree, and every time you go
  32     : //! left, you add a `0` to the code, and every time you go right, you add a `1`.
  33     : //!
  34     : //! Given a list of symbols (like words, sentences, whatever we want to assign
  35     : //! our codes to) and their frequencies, we can build a Huffman tree from which
  36     : //! each symbols Huffman code can be deduced. Then, we simply replace the
  37     : //! symbols in our original text with their respective codes, and we now have a
  38     : //! shorter message.
  39     : //!
  40     : //! To decode it, you can simply start reading the bit-stream, and eagerly match
  41     : //! on any Huffman codes you find. Since only the leaf nodes of the tree
  42     : //! contain symbols, you will never be in an ambigous situtation.
  43     : //!
  44     : //! There are additional caveats to the specific form of Huffman coding used by
  45     : //! the RFC, but it's mostly just minor details.
  46     : //!
  47     : //! ### LZ77 algorithm
  48     : //!
  49     : //! This algoritm takes a different approach: it only focuses on eliminating as
  50     : //! many repetitions as it can. When scanning the text, it keeps a history of
  51     : //! the previously parsed text, and every repetition, it replaces with a sort of
  52     : //! pointer, indicating how far back to look in the history, and how many
  53     : //! symbols to repeat.
  54     : //!
  55     : //! For example, lets say we wanted to decompress the following input:
  56     : //!
  57     : //! ```text
  58     : //! criss [D=6,L=2]o[D=6,L=2] apple sauce
  59     : //! ```
  60     : //!
  61     : //! When we see a `[D,L]` pair, it indicates the Distance we must go backward
  62     : //! in the stream and the Length of the data we must copy. The above example
  63     : //! decodes to:
  64     : //!
  65     : //! ```text
  66     : //! criss cross apple sauce
  67     : //! ```
  68     : //!
  69     : //! The `cr` and `ss` parts are repeated between `criss` and `cross`. Of
  70     : //! course, this compression only makes sense if we copy a large amount of data,
  71     : //! not just two characters.
  72     : //!
  73     : //! ### DEFLATE, in detail
  74     : //!
  75     : //! DEFLATE works by combining these two algorithms: first it eliminates
  76     : //! duplicates with LZ77, then uses Huffman coding to shrink the input down even

  78     : //!
  79     : //! * No compression at all (mainly used for wrapping already compressed data in a GZIP header format)
